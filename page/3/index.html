关于本博客及这些日志的说明

发表于 2017-09-11 | 分类于 博客声明

简介

  首先，我个人是非常喜欢linux系统，工作最初是从事.net开发，接触最多的是windows平台下的iis应用服务器，后来由于部门变动，被调岗至云平台部门，由于部门需要，所以转到了php开发。然后一干就是3年多吧，也是从那时候开始接触linux。
  最早接触的发行版是ubuntu，工作中开发环境一直都是unbuntu。
  最开始从windows开发转向linux开发，慢慢的适应下来，并且就喜欢上了linux，随后开始一点点了解linux的起源、历史、各种命令、操作…

自我评价

  才疏学浅、能力有限！写了这么多博客不代表我真的能对linux多么精通，甚至可以说皮毛都触摸不到。linux博大精深，我连其内核源码都没有认真看过(实在是精力和能力有限，惭愧…)，何敢称精通呢，绝非妄自菲薄…
  虽然写了不少服务器架构等日志，也不代表我对这些高负载高并发高可用等等的各种架构有多么擅长和精通，也只敢说是接触了皮毛，而且并没有经过超高并发的考验。更何况，我认为，一个成熟的系统，决然不是某个人或某几个人的努力成果，必然是一个团队经过多年的沉淀和积累，踩过一个个大大小小的坑,经过大量用户高并发高负载的考验逐年打磨出来的。而且技术永远是为业务服务的，业务需求推动技术发展和团队进步…
  虽然才疏学浅，但这不妨碍我想穿越计算机和网络迷雾的决心和欲望，不影响我想写出高性能代码的追求。也正是出于此原因，才有了这一系列的博客文章。
  这些文章绝大部分都是自己总结而来，也有不少是参考其他一些大神的博客和日志，也看了很多linux相关视频和书籍，再加上这几年来工作中所用，将这些整理成一篇篇日志，想将自己所学尽可能系统的记录下来，鞭策自己不断进步，鞭策自己不断的深层次探究，鞭策自己尽可能弄清楚整个WEB应用的各个环节。

写博客的目的

  最初写这个博客的目的是想要梳理自己这几年来对linux的一点点积累，因为平时所学所用难免会比较零碎，不成体系，而且时间长了难免会遗忘。
  我本人对linux的认知和掌握，有的来源于工作中，有的来源于自己学习摸索的，也有的来源于看相关linux视频、书籍以及别的大神写的博客，而且平时为了防止遗忘，所以会记些笔记。慢慢的就有了将这些知识好好梳理一遍，并写成博客、日志的想法，也算是相当于将之前所学再次系统的回顾一遍。
  在写这些博客的时候，才真的感受到自己技术的缺憾和文笔的匮乏，真想把某些深层次机制讲解透彻，真的很难，首先自己一定要做到刨根问底的理解。如果自己都云里雾里，那别人看了岂不是会被误导(虽然可能由于我个人的理解浅薄已经误导了…不过还好，github上这个博客互联网一般搜不到…),而且很多时候不仅要描述得到，排版精美，还要配上一些图片辅助理解…
  真想做到尽善尽美的时候，发现自己是心有余而力不足，再加上本来github的博客在百度里面也一般搜不到，所以慢慢的将目的从“分享知识”到“审视自己”。也就是说通过将自己所学，写成博客，并借此鞭策自己尽可能的搞透彻这些技术点，做到对自己博客中的每一篇都了如指掌，并且如果随着时间长了有遗忘的话，还能找到这些文章再去研究。

  这本身就是一种“知识沉淀”和“自我审视”的过程，也是一种查缺补漏的过程，自己的知识体系也能慢慢的打扎实，又能在互联网上留下一些痕迹，何乐而不为呢？

深层次原因

  当用户在键盘上输入http://www.taobao.com，到浏览器显示完整网页，这期间到底发生了什么？

  假设已经建立好了tcp连接通道，从用户浏览器将这个请求封装成http报文，再经过底层的tcp、ip报文封装，再经过用户电脑网卡的链路层报文封装，最后经过物理层的有线设备或无线设备转换成能在现实世界中传输的电磁信号，经过层层交换机和路由器，经过DNS解析、CDN网络转发等等的一路坎坷，最终到达的却是一个缓存服务器…然后从内存中就直接返回了请求结果。
  连淘宝的数据库都没摸到，更何谈让淘宝的服务器转动他那磁盘进行寻道了。
  假设同时有几十万个用户同时敲下了这个命令，四面八方的tcp请求疯狂涌来，这些请求到底是被怎么调度和处理的？每一个请求到达不同的服务器之后，又是一个什么样的形式？每一个请求大概又会占用服务器多少的内存和带宽？同一台服务器内，这么多用户请求建立的进程(或线程)之间又是怎么相互调度运转的？他们之间能共享服务器内存中的某些数据吗？到底怎么能让一台服务器面对上万甚至更高的并发做到从容不迫、有条不紊？

  这一路下来，有太多的疑问和知识点，可以说对于WEB开发人员，使用PHP等高级语言，不关心底层某些特性，不清楚网络请求报文是如何在服务端和客户端进行流转，不清楚自己的哪些代码对服务器的CPU和内存造成了多大的困扰的话，在一层层被迷雾包裹住的虚拟环境中写着自己的代码，很难保证网站性能。
  每一个静态资源的请求，都是一个tcp连接，一个html页面中，引用了大量的js、css、图片等静态资源，到底会对服务器造成多大的性能压力和带宽压力？很多网站都运行了很多年了，或者由于用户量极大，所以数据库数据量也极大，那么这种情况下，sql查询就要慎之又慎，动辄百万条、千万条的数据查询，再加上各表的join等操作，系统疯狂的转动存储磁盘，这时候就不得不探究数据库的各种优化，索引、缓存、分表、分库、分片…

  随着用户量越来越大，并发越来越高，就一台服务器越来越力不从心，慢慢增加服务器数量，各种缓存服务器、sql服务器、负载均衡等等等架构的出现以及各种nosql和消息队列的出现，WEB开发人员难道不应该熟知自己公司的架构，并且在代码层面去积极配合这些架构吗？程序员不得不懂运维！

  想要写出极致性能的代码，想要设计出极致性能并且高可用的架构，想要写出适配所有终端及各版本浏览器的页面，想要用户获得最爽快的访问体验，这一切真的很难，而这也是我所追求的，虽然还差的很远…

  由于本人水平实在有限，而且不是专门搞底层、驱动及内核开发的，也不是C语言和汇编语言出身(虽然大学时候经常倒腾C语言、反汇编、游戏外挂，但现在由于工作和这些实在打不上边，慢慢就还回去了= =！)，所从事的工作也大部分都是写逻辑、写服务、写页面，小到各浏览器对某个浏览器的样式兼容，再到各终端及屏幕尺寸的响应式适配，再到服务端业务逻辑处理，再到数据库端查询优化、缓存构建，再到部门的服务器基础运维。

  工作所限，精力所限，能力有限，很想将WEB这一行上面每一个细节点探究透彻，然而才疏学浅，差的太多。有时候在写某些原理性日志时，恨不得去再钻研C语言，将整个linux内核吃透，然而这起码目前来说只能是痴人说梦。只能旁征博引，尽可能深的挖掘内部原理，又不触碰底线，最起码做到自己心中有数，自己能将自己说服，能把整个知识体系给闭合就足够日常开发了，只能等待日后有机会再去深入学习底层语言，从源码角度研究linux内核了…

  虽然不是做底层驱动和内核出身，虽然不是C语言和汇编出身，但这并不妨碍我想要穿过计算机迷雾，探究整个WEB访问从报文传输到服务器进程处理，再到缓存及数据查询优化的欲望和决心…

  缓存，sql调优，系统调优和检测，负载均衡，高可用，微服务架构，数据备份与恢复，站点日志分析、查询与统计等等的一系列所必备的知识，不管是工作中用到的还是没用到的，不管是自己摸索的还是通过其他途径学习的，这些都是对我自己的要求,也都是这个博客里面所要写清楚，也是我所必须要深刻掌握和理解的。我始终认为，只有拨开这一系列的迷雾，站在整个架构的高度上，写出来的代码才是合格的代码。

最后

  瞎写了一大堆废话，算是给自己一个交代，给自己一个总结，给自己一个鞭策，时刻提醒自己还有很多路要走，还有太多东西要学…

linux系统基础命令(一)

发表于 2017-09-10 | 分类于 linux基础

关于终端

linux系统中，一切皆文件，哪怕是设备。终端设备也在文件中有所体现

  1. 控制台终端(console):

    早期由于计算机特别贵，而且unix是多人多任务操作系统，
所以为了能让多人同时使用unix进行工作,所以就采用比较便宜的物理终端接入unix
控制台终端所使用文件/dev/console

  2. 虚拟终端(tty)：

    linux系统启动时，会自动启动6个虚拟终端
这6个虚拟终端之间可以使用ctrl+alt+F1-6之间切换
使用tty查看当前使用的终端。
tty
#/dev/tty1
#输出结果和当前使用的是哪个终端相关，比如ctrl+alt+F2,则显示/dev/tty2,其他同理

  3. 伪终端(pts):

    一般通过ssh远程连接登录的终端就属于伪终端，使用tty命令查看：
#/dev/pts/1
#如果同时再有一个ssh连接到本系统，则显示/dev/pts/2
#ls /dev/pts/   此命令可查看当前连入系统的有哪几个虚拟终端

关于命令

linux系统中有两种命令，内建命令、外部命令，使用type命令可查看命令类型

  1. 内建命令：
    由shell自带
  2. 外部命令
    当前系统的文件系统某路径下有对应执行文件
which和whereis可查看所在路径
shell按照PATH环境变量所指定的路径自左向右的依次搜寻
echo $PATH 查看PATH环境变量
只有第一次才会去搜寻路径，一旦搜寻到之后，会缓存至一个内存中的key-value存储中
使用hash可查看该缓存
hits    command         #hits为缓存命中次数
5        /bin/ls
如果hash中缓存的程序路径发生改变，比如mv /bin/ls /usr/bin/ls，则会找不到ls命令
此时可以使用hash -r 清空缓存
  3. 命令行接口： 
    [root@hostname ~]# command [options] [arguments]
    root : 当前登录的用户
    hostname : 当前主机名，显示的并非完整主机名
    ~ : 当前处于的目录，也称工作目录，此处显示的是相对路径，使用pwd命令可显示完整目录
    # : 命令提示符，其中#代表root用户；$代表普通用户 
    options: 有短选项如 -l -h -c等，短选项一般可合并，如-lhc
                有长选项如 --help等，长选项一般不能合并
    arguments: 命令的作用对象，多个参数之间一般用空格隔开        
    ctrl+c可取消命令执行

linux文件系统

  1. 文件名命名规范
    文件名严格区分大小写
文件名可以使用除/之外的任意字符
所有.开头的文件都是隐藏文件
文件名长度不能超过255个字符

  1. 内建命令:
    help command
  2. 外部命令:
    (1) command --help
    command -h
(2) 使用手册 manual   
    man <command>
(3) 信息页
    info <command>    
(4) 程序自带的帮助文档

history命令

linux系统中，所有命令历史都保存在内存中。当用户登出时，系统会将内存中的命令历史追加写入当前用户家目录的.bash_history文件中

  1. 常用选项
    history -a      #追加本次登录后使用的命令至.bash_history文件中
history -d 40   #删除指定编号的历史命令
history -c      #清空命令历史
history 10      #显示历史中最近的10条命令；
  2. 快捷操作
    !40             #调用历史中指定编号的命令
!str            #调用历史中最近一次包含str的命令
!!              #调用上一次命令
ESC然后按.       #调用上次命令的最后一个参数
  3. 环境变量

man命令

  1. 简介

    man手册所在目录为：/usr/share/man  
不同作用的命令存在不同的目录
man命令会去对应的目录下找到.gz格式的压缩文件进行解压并用less命令展示结果
有些命令存在于多个章节中，whatis <command>可以查看该命令在哪个章节
使用 man num <command> 可以查看第num章节的帮助文档

  2. 章节介绍

    man1            #user commands   普通用户命令  
man2            #system call     系统调用
man3            #c库调用
man4            #设备文件及特殊文件
man5            #配置文件格式
man6            #游戏相关
man7            #杂项
man8            #管理类命令

  3. man命令配置文件

    配置文件：       /etc/man.config  这里面指明去哪搜索man文档

  4. man命令段落说明

    NAME            #功能性说明
SYNOPSIS        #语法格式
                []：可选内容；
                <>：必须提供的内容；
                a|b|c：多选一；
                ...：同类内容可出现多个；
DESCRIPTION     #描述
OPTIONS         #选项
EXAMPLES        #示例
  5. 操作方法
    空格键           #向文件尾翻一屏；
b               #向文件首部翻一屏；
Ctrl+d          #向文件尾部翻半屏；
Ctrl+u          #向文件首部翻半屏；
回车键           #向文件尾部翻一行；
k               #向文件首部翻一行；
G               #跳转至最后一行；
2G              #跳转至第2行；

  6. 文本搜索

    /keyword        #从文件首部向文件尾部依次查找；不区分字符大小写；
?keyword        #从文件尾部向文件首部依次查找；    
n               #与查找命令方向相同；
N               #与查找命令方向相反；

  7. 退出

    q               #quit
